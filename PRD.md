당신은 시니어 풀스택 개발자이자 제품 엔지니어입니다.
지금부터 7일 개발 대회 출품용 웹서비스 **"여담(女談)"**의 웹을 개발해 주세요.
이 프로젝트는 단순한 화면 목업이 아니라, 회원가입부터 제공자 등록, 요청글 작성, 카테고리 필터링, 연락처 확인까지 실제로 동작해야 하는 웹 페이지를만들거야.

# 1. 프로젝트 개요
## 서비스명
여담(女談)
## 슬로건
경력이 없어도, 경험은 있습니다.
## 한 줄 정의
경력으로 인정받지 못했던 여성의 생활 경험을, 그 경험이 필요한 여성과 연결하는 생활 경험 매칭 게시판 서비스
## 서비스 목적
여담은 간병 경험, 살림, 정리수납, 반려동물 돌봄, 요리, 스마트폰 사용 도움 등 여성들이 생활 속에서 쌓아온 경험을 유상 서비스로 연결할 수 있도록 돕는 웹서비스입니다.
제공자는 자신의 경험과 가능한 서비스를 프로필로 등록하고, 이용자는 필요한 도움을 제공자 목록에서 찾거나 요청글을 직접 작성할 수 있습니다.
여담은 제공자를 직접 고용하거나 서비스를 직접 제공하는 업체가 아니라, 이용자와 제공자가 서로를 발견하고 직접 연락할 수 있도록 돕는 게시판형 연결 플랫폼입니다.

다음 기술을 사용해 주세요.
* Next.js 최신 안정 버전
* App Router

* 직접 작성한 간단한 SVG
무거운 상태 관리 라이브러리는 사용하지 않습니다.

5. 기존 구현이 있다면 재사용 가능한 부분을 구분합니다.
6. 현재 상태를 간단히 요약합니다.
7. 구현 순서를 제안합니다.

기존 프로젝트가 없다면 Next.js App Router, TypeScript, Tailwind CSS 기반으로 프로젝트를 구성해 주세요.

# 5. 회원 및 권한 원칙
회원가입 시 이용자와 제공자를 고정적으로 구분하지 않습니다.
모든 가입 회원은 기본적으로 다음 기능을 사용할 수 있습니다.
* 제공자 프로필 등록
* 요청글 작성
* 다른 제공자 프로필 열람
* 다른 요청글 열람
* 로그인 후 연락처 확인
제공자 여부는 `provider_profiles` 테이블에 본인의 프로필이 존재하는지로 판단합니다.
시스템 역할은 다음 두 개만 사용합니다.
* user
* admin
MVP에서는 관리자 전용 화면을 구현하지 않아도 됩니다.

# 10. Supabase 데이터 모델
다음 SQL 스키마를 기준으로 구현하되, 실제 프로젝트 상황에 맞게 타입과 제약조건을 보완해 주세요.
## profiles
```text
id uuid primary key references auth.users(id)
name text not null
nickname text not null
phone text not null
avatar_url text null
role text not null default 'user'
created_at timestamptz default now()
updated_at timestamptz default now()
```
비밀번호는 profiles에 저장하지 않습니다.
## provider_profiles
```text
id uuid primary key
user_id uuid unique not null references profiles(id)
headline text not null
category_id uuid not null references categories(id)
experience_years integer null
experience_description text not null
introduction text not null
available_days text[] null
available_time_text text null
available_services text null
unavailable_services text null
contact_method text not null
contact_value text not null
contact_available_time text null
status text not null default 'ACTIVE'
created_at timestamptz default now()
updated_at timestamptz default now()
```
status 허용값:
```text
ACTIVE
PAUSED
```
## categories
```text
id uuid primary key
name text not null
slug text unique not null
icon text null
description text null
sort_order integer not null default 0
is_featured boolean not null default false
is_active boolean not null default true
```
## service_items
```text
id uuid primary key
category_id uuid not null references categories(id)
name text not null
slug text not null
sort_order integer not null default 0
is_active boolean not null default true
```
## provider_service_items
```text
provider_profile_id uuid references provider_profiles(id)
service_item_id uuid references service_items(id)
primary key(provider_profile_id, service_item_id)
```
## request_posts
```text
id uuid primary key
user_id uuid not null references profiles(id)
category_id uuid not null references categories(id)
service_item_id uuid null references service_items(id)
title text not null
content text not null
desired_date date not null
desired_time text null
duration_text text null
budget_text text null
is_negotiable boolean not null default false
is_urgent boolean not null default false
contact_method text not null
contact_value text not null
status text not null default 'OPEN'
created_at timestamptz default now()
updated_at timestamptz default now()
```
status 허용값:
```text
OPEN
CLOSED
```
## reports
```text
id uuid primary key
reporter_user_id uuid not null references profiles(id)
target_type text not null
target_id uuid not null
reason text not null
description text null
status text not null default 'PENDING'
created_at timestamptz default now()
```
target_type 허용값:
```text
provider
request
```
status 허용값:
```text
PENDING
REVIEWED
```

3. 사용자에게 실행 방법을 문서화

# 12. RLS 정책
## profiles
* 로그인 사용자는 자신의 프로필 조회 및 수정 가능
* 공개 화면에 필요한 닉네임 정도만 다른 테이블 조인을 통해 표시
* 다른 사용자의 전화번호와 이메일을 직접 조회하지 못하게 처리
## provider_profiles
* ACTIVE 상태는 공개 조회 가능
* PAUSED 상태는 소유자만 조회 가능
* 로그인 사용자는 자신의 제공자 프로필 등록 가능
* 수정 및 삭제는 소유자만 가능
* 한 사용자당 하나의 프로필만 생성 가능
## provider_service_items
* 공개 제공자에 연결된 항목은 조회 가능
* 추가 및 삭제는 제공자 프로필 소유자만 가능
## request_posts
* 목록과 상세는 공개 조회 가능
* 등록은 로그인 사용자만 가능
* 수정 및 마감은 작성자만 가능
* 연락처 공개는 별도 서버 로직을 사용해 제한
## reports
* 로그인 사용자만 신고 가능
* 신고자는 자신의 신고만 조회 가능
* 다른 일반 사용자의 신고는 조회 불가
* admin만 전체 신고 조회 가능

# 13. 연락처 보호 방식
연락처를 공개 목록 쿼리에 포함하지 마세요.
목록 화면에서는 연락처 필드를 조회하지 않습니다.
상세 화면의 최초 상태에서는 다음처럼 마스킹합니다.
```text
010-12**-56**
```
연락처 전체 공개 조건:
1. 로그인 회원
2. 안전 안내 모달 확인
3. 대상이 활동중 또는 모집중 상태
4. 서버 측에서 사용자 세션 검증
가능하면 Next.js Server Action, Route Handler 또는 Supabase RPC를 사용해 연락처를 반환해 주세요.
클라이언트에 이미 전체 연락처를 내려보낸 뒤 CSS로 가리는 방식은 사용하지 마세요.
안전 안내 문구:
```text
여담은 이용자 간 직접 연결을 돕는 게시판형 서비스입니다.
첫 만남은 공개된 장소에서 진행하고,
선입금이나 과도한 개인정보 요구에 주의해 주세요.
```
비회원이 연락처 확인 버튼을 누르면 로그인 화면으로 이동합니다.

# 20. 목업 데이터
데모가 비어 보이지 않도록 목업 데이터를 준비해 주세요.
최소 기준:
* 제공자 프로필 12개
* 요청글 12개
* 긴급 요청글 2개 이상
* 마감 요청글 3개 이상
* 대표 카테고리별 제공자 2명 이상
* 대표 카테고리별 요청글 2개 이상
* ACTIVE 제공자 10명
* PAUSED 제공자 2명
대표 분배:
* 병원·외출 동행: 제공자 3명, 요청글 3개
* 반려동물 돌봄: 제공자 2명, 요청글 2개
* 집안일 도움: 제공자 2명, 요청글 2개
* 정리수납: 제공자 2명, 요청글 2개
* 디지털 생활 도움: 제공자 3명, 요청글 3개
목업 문구는 서로 중복되지 않도록 작성합니다.
예시 프로필 제목:

# 26. 금지 사항
다음 방식은 사용하지 마세요.
* 연락처 전체를 클라이언트에 내려보낸 뒤 CSS로 숨기기
* RLS 없이 프론트엔드 버튼만 숨겨 권한 처리
* service role key를 브라우저 코드에 사용
* 비밀번호를 profiles 테이블에 저장
* 카테고리를 여러 화면에 중복 하드코딩
* 존재하지 않는 기능을 화면에서 작동하는 것처럼 표시
* 신고를 저장하지 않고 "신고 접수 완료"라고 표시
* 사용자가 이해하기 어려운 영어 오류 메시지를 그대로 노출
* `any` 타입 남용
* 불필요한 전역 상태 관리
* 거대한 단일 컴포넌트
* 사용되지 않는 패키지 설치
* 빌드 오류를 남긴 상태로 작업 완료 처리
* 의료행위나 전문 간병을 제공 가능한 서비스처럼 표현
* 본인인증이 없는데 검증된 이용자라고 표현
